## Containerizing cc-fleet: Bringing a Go CLI for LLM orchestration to OpenShift

Developer tools don't have to be server-side to benefit from containerization. In this post, we walk through containerizing and validating cc-fleet, a Go CLI that orchestrates third-party LLM providers for Claude Code's multi-agent workflows, on Red Hat OpenShift using UBI-based container images.

## What is cc-fleet?

cc-fleet is a Go command-line tool that solves a real problem for developers using AI coding agents: it lets any third-party LLM provider (DeepSeek, Qwen, GLM, and others) plug into Claude Code's Dynamic Workflows, Agent Teams, and Subagents. Without cc-fleet, these multi-agent features only work with Anthropic's own models.

The tool manages provider profiles, handles API key routing through pluggable secret backends, and can spawn teammate sessions in tmux windows. It's a substantial codebase: 391 Go source files organized across 30+ internal packages, with a Cobra CLI framework and BubbleTea TUI interface.

## Why containerize a CLI tool?

Most containerization discussions focus on web services and APIs, but CLI tools have compelling reasons to live in containers too:

- **Reproducible distribution:** A container image guarantees the binary runs identically across environments, with all dependencies satisfied.
- **CI/CD integration:** Containerized CLI tools can be used as pipeline steps in Tekton or GitHub Actions without installing language toolchains.
- **Validation at scale:** Running CLI commands as Kubernetes Jobs lets you test tools across different cluster configurations systematically.
- **Security scanning:** Container images go through standardized vulnerability scanning pipelines.

For cc-fleet specifically, containerizing the binary proves that the Go compilation produces a clean, portable artifact suitable for enterprise distribution channels.

## Building with UBI multi-stage Dockerfiles

We used a two-stage build approach:

```dockerfile
# Stage 1: Build with Go toolset
FROM registry.access.redhat.com/ubi9/go-toolset AS builder
WORKDIR /opt/app-root/src
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -ldflags="-s -w" -o /opt/app-root/cc-fleet ./cmd/cc-fleet

# Stage 2: Minimal runtime
FROM registry.access.redhat.com/ubi9/ubi-minimal
COPY --from=builder /opt/app-root/cc-fleet /usr/local/bin/cc-fleet
RUN microdnf install -y shadow-utils && microdnf clean all && \
    chgrp -R 0 /usr/local/bin/cc-fleet && chmod -R g=u /usr/local/bin/cc-fleet
USER 1001
ENTRYPOINT ["cc-fleet"]
CMD ["--help"]
```

Key decisions in this Dockerfile:

- **`CGO_ENABLED=0`** produces a fully static binary with no external C library dependencies. This is critical for the ubi-minimal runtime stage, which doesn't include glibc development headers.
- **`-ldflags="-s -w"`** strips debug symbols, reducing the binary size.
- **`chgrp -R 0` and `chmod -R g=u`** satisfy OpenShift's arbitrary UID requirement. OpenShift assigns random UIDs in group 0, so the binary must be readable and executable by that group.
- **`USER 1001`** ensures non-root execution.
- **`ubi-minimal`** keeps the runtime image small: just the binary on top of a minimal RHEL base.

## OpenShift binary builds: from source to registry

Instead of building locally with Podman, we used OpenShift's native binary build strategy:

```bash
# Create a BuildConfig
oc new-build --name="cc-fleet-cc-fleet" \
  --binary --strategy=docker \
  --to-docker --to="quay.io/aicatalyst/cc-fleet:latest" \
  --push-secret=autopoc-registry-push \
  -n autopoc-test-builds

# Upload source and build on-cluster
oc start-build cc-fleet-cc-fleet \
  --from-dir=./repos/cc-fleet/ \
  --follow --wait \
  -n autopoc-test-builds
```

The cluster pulls the UBI base images, compiles the Go binary, and pushes the result directly to Quay.io. No local container runtime needed. The entire build took about two minutes.

## Job-based deployment for CLI validation

Since cc-fleet is a CLI tool, not a web service, deploying it as a Kubernetes Deployment would cause CrashLoopBackOff: the container runs a command and exits. Instead, we used Kubernetes Jobs, one per test scenario:

```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: cc-fleet-help
  namespace: poc-cc-fleet
spec:
  backoffLimit: 1
  activeDeadlineSeconds: 60
  template:
    spec:
      containers:
        - name: cc-fleet
          image: quay.io/aicatalyst/cc-fleet:latest
          command: ["cc-fleet"]
          args: ["--help"]
          securityContext:
            allowPrivilegeEscalation: false
            capabilities:
              drop: ["ALL"]
      restartPolicy: Never
```

This pattern works well for any CLI tool validation: define the command and arguments, run it as a Job, check the logs for expected output.

## Test results and what we learned

We ran two validation scenarios:

| Scenario | Result | Details |
|---|---|---|
| Help output | Pass | Displayed all 30+ available commands with descriptions |
| Doctor diagnostics | Pass | Ran 10 health checks; correctly reported missing Claude Code and tmux |

The doctor command was particularly interesting. It ran its full diagnostic suite inside the container and accurately reported what was missing: no Claude Code CLI, no tmux, no settings files. This confirms the binary handles missing dependencies gracefully rather than crashing.

The help output showed the complete command surface area: provider management (add, edit, remove, list), session control (run, spawn, teardown), and multi-agent orchestration (subagent, workflow, team). All 30+ commands rendered correctly in the container environment.

## What we learned

**Go CLI tools are ideal containerization candidates.** With `CGO_ENABLED=0`, the binary is completely self-contained. The multi-stage build pattern (go-toolset builder plus ubi-minimal runtime) is a reliable template for any Go project.

**Job-based testing catches real issues.** The original PoC plan included a `version` test scenario with `cc-fleet version`, which failed because version is a flag (`--version`), not a subcommand. Kubernetes Jobs surfaced this quickly through exit codes.

**Desktop tools provide diagnostic value even in containers.** cc-fleet's doctor command, designed for checking developer workstation setup, still produced useful output in the container. The structured diagnostic format (check name, status, hint) could inform how similar tools report environment status in platform contexts.

## Try it yourself

The container image is available at `quay.io/aicatalyst/cc-fleet:latest`. To run cc-fleet in your own cluster:

```bash
kubectl run cc-fleet-test --image=quay.io/aicatalyst/cc-fleet:latest \
  --restart=Never -- --help
kubectl logs cc-fleet-test
```
